### 결정 사항
- 버전 체계는 SemVer(`MAJOR.MINOR.PATCH`) 사용
- 기능 개발과 문서(`CHANGELOG`/`DEVLOG`)를 병행 업데이트
- 원격 기본 브랜치(`main`)는 직접 덮어쓰지 않고 브랜치/PR로 반영
- 신규 게임은 포털 내 로컬 URL로 바로 플레이 가능하게 우선 통합
- 모바일 조작 입력은 예측 가능성이 높은 버튼 기반 방식을 우선 채택
- 시작 직후의 억울한 피격을 줄이기 위해 카운트다운+보호구간을 기본 규칙으로 채택
- 라이브러리는 경량 우선 원칙으로, 실효성이 높은 Howler.js부터 채택
- 난이도는 하드코딩 분기보다 프리셋 파라미터 기반으로 관리
- 사용자 입력이 섞이는 렌더링은 `innerHTML` 대신 DOM API를 기본으로 사용
- 패턴 다양화는 난이도/점수 스케일에 따라 점진적으로 강화
- 게임 스크립트는 단일 파일보다 기능 계층 모듈 분리를 우선
- 신규 내장 게임도 초기 단계부터 모듈 구조를 적용
- Lane Switch는 단계적 확장(미션 → 아이템 → 설정) 방식으로 리스크를 낮춰 개발
- 아이템은 "짧은 효과 + 명확한 피드백" 원칙으로 밸런싱
- 확장 아이템은 보상형(자석/더블) + 리스크형(오버드라이브) 조합으로 설계
- 모바일 포털은 정보 밀도 우선(2열 카드)을 기본 전략으로 유지
- 관리자 입력은 포털 렌더 필드와 1:1 매핑(미리보기/스크린샷)하도록 확장
- 미디어 입력값은 저장 전 정규화하여 렌더 안정성을 우선 확보
- 모바일 포털은 핵심 탐색(게임 카드)이 최근 업데이트보다 우선 노출되도록 유지
