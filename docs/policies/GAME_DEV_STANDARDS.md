# GAME_DEV_STANDARDS

본 문서는 앞으로 모든 게임 개발에 적용되는 공통 기준입니다.

## 1) 품질 기준

- "15년차 베테랑 게임 개발자" 관점으로 설계/구현/테스트
- 플레이 루프(핵심 재미), 조작 반응성, 피드백, 밸런스, 안정성 우선
- MVP라도 확장 가능한 구조를 유지

## 2) 아키텍처 기준 (모듈화)

- 단일 거대 스크립트 금지
- 책임 분리 원칙:
  - Core: 게임 상태/룰/시뮬레이션
  - Feature: 기능 단위 시스템(전투, 이동, 경제 등)
  - UI: HUD/메뉴/알림
  - Infra: 저장, 설정, 로깅, 플랫폼 연동
- 파일 분할 기준:
  - "기능 1개 + 책임 1개" 원칙
  - 파일이 커지면(예: 250~400줄 이상) 하위 모듈 분리 검토

## 3) 확장성 기준

- 하드코딩 최소화(상수/밸런스/테이블 데이터 분리)
- 씬과 로직 결합도 최소화
- 이벤트/신호 기반 연결 우선
- 테스트 가능한 단위(순수 로직) 분리

## 4) 문서화 기준

게임별로 아래 문서를 반드시 유지:
- 00_game_blueprint.md
- 01_gdd.md
- 02_development_plan.md
- 03_roadmap.md
- 04_devlog.md
- 05_qa_checklist.md
- 06_release_notes.md

## 5) 에셋 운용 기준

- 필요한 에셋(이미지/오디오/UI)은 개발 진행을 위해 에이전트가 선제적으로 확보 가능
- 저장 위치는 C 드라이브를 피하고 D 드라이브 우선 사용
  - 기본 경로: `/mnt/d/OpenClaw_Downloads/game-assets/`
- 프로젝트 반영 시 라이선스/출처를 문서(`04_devlog`, `06_release_notes`)에 함께 기록
- 대용량 원본은 D 드라이브에 보관하고, 프로젝트에는 최적화본만 포함

## 6) Git 동기화 기준

- 기본 원칙: 로컬 우선 개발
- 원격(GitHub) 업로드는 마일스톤/의미 있는 완성 단계에서만 수행
- 매 작업 단위 push/PR 생성은 지양
- 필요 시점(사용자 승인 또는 릴리즈/리뷰 타이밍)에 브랜치 정리 후 일괄 반영

## 7) 진행/보고 기준

- 작업 시작 전: 목표/범위/다음 단계 공유
- 장기 작업: 10분 간격 또는 단계별 중간 보고
- 완료 보고: 결과 + 검증 + 리스크 + 다음 액션
