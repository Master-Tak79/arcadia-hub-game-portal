# 07_dev_guidelines — Godot Relic Survivor

## 목적

본 문서는 `Godot Relic Survivor` 개발 시 코드 품질, 구조, 검증, 문서화 기준을 프로젝트 단위로 고정합니다.

## 1) 아키텍처 원칙

- 단일 거대 스크립트 금지
- 기능/책임별 모듈 분리 유지
  - `core`: 상태, 루프, 입력, 공통 시그널
  - `entities`: 플레이어/적/투사체/보스
  - `systems`: 전투, 스폰, 경험치, 업그레이드, 웨이브
  - `ui`: HUD, 레벨업 패널, 일시정지, 게임오버
  - `data`: 밸런스/테이블/업그레이드 데이터

## 2) 코드 규칙

- 하드코딩 최소화, 밸런스 수치는 `scripts/data/*`로 분리
- 신기능은 "시스템 + 데이터 + UI 반영" 3축으로 구현
- 파일 300줄 이상 시 분리 우선 검토
- 이벤트 흐름은 직접 참조보다 신호/시스템 경유 우선

## 3) 전투/밸런스 규칙

- 코어 루프: 이동 → 자동공격 → 처치/성장 → 웨이브 대응
- 초반 3분은 학습 구간, 4~8분 빌드 체감, 9~12분 압박 구간
- 신규 적/업그레이드는 기존 루프를 깨지 않도록 단계적 도입

## 4) 검증 규칙

- 모든 단계에서 최소 1회 `headless` 스모크
- 주요 변경 후 `fixed-fps` 장시간 시뮬레이션 수행
- 기능 완료 시 QA 체크리스트 즉시 갱신

## 5) 문서 규칙

- 설계 변경 즉시 `01_gdd.md`, `02_development_plan.md` 동기화
- 작업 결과 즉시 `04_devlog.md`, `06_release_notes.md` 반영
- 완료 기준/리스크는 항상 문서에 명시

## 6) 에셋 규칙

- 에셋 원본은 D 드라이브 우선: `/mnt/d/OpenClaw_Downloads/game-assets/`
- 프로젝트 반영본은 최적화 후 저장
- 출처/라이선스/수정 여부를 개발일지 또는 릴리즈노트에 기록
